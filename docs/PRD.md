## 웹 페이지 및 문서 썸네일 생성기 PRD (제품 요구사항 명세서)

안녕하세요! 웹사이트(URL)나 다양한 형식의 파일(HTML, PDF, DOC, MD)을 이미지로 변환해 주는 웹 기반 도구 개발을 위한 제품 요구사항 명세서(PRD)입니다. 이 문서는 주니어 개발자분들도 쉽게 이해하고 개발을 시작할 수 있도록 상세하고 친절하게 작성되었습니다.

---

### 1. 제품 개요 (Introduction)

#### 1.1. 제품명
**Thumblify** (썸블리파이) - 'Thumbnail' (썸네일)과 '-ify' (~화하다)의 합성어로, 무엇이든 썸네일 이미지로 만들어준다는 의미를 가집니다.

#### 1.2. 배경 및 문제 정의 (Problem Statement)
콘텐츠를 공유하거나 관리할 때, 링크나 파일 이름만으로는 내용을 파악하기 어렵습니다. 예를 들어, 여러 개의 URL을 저장해두었을 때 각 사이트가 어떤 모습이었는지 일일이 접속하지 않고는 알 수 없습니다. PDF나 워드 문서도 마찬가지입니다. 이러한 불편함을 해결하기 위해, **웹사이트 URL이나 문서 파일을 입력하면 해당 콘텐츠의 첫 페이지를 시각적으로 명확하게 보여주는 이미지(썸네IL)를 즉시 생성해주는** 간단하고 직관적인 웹 도구가 필요합니다.

#### 1.3. 목표 (Goals)
* **사용자 목표:** 누구나 쉽게 URL이나 파일을 이미지로 변환하여 콘텐츠를 한눈에 파악하고 편리하게 공유 및 관리할 수 있도록 한다.
* **비즈니스 목표 (초기):** 개발 및 유지보수 비용을 최소화하여 무료로 서비스를 제공하고, 개발자의 포트폴리오 강화 및 잠재적 사용자층 확보에 집중한다. 향후 API 제공을 통한 수익 모델을 고려할 수 있다.

#### 1.4. 타겟 사용자 (Target Audience)
* **콘텐츠 큐레이터/마케터:** 블로그나 SNS에 공유할 링크의 미리보기 이미지를 빠르고 예쁘게 만들고 싶은 사람.
* **웹 개발자/디자이너:** 자신이 만든 웹사이트나 포트폴리오의 썸네일을 생성하여 공유하고 싶은 사람.
* **학생/연구원:** 참고 자료(PDF, DOC)의 첫 페이지를 이미지로 만들어 시각적으로 분류하고 관리하고 싶은 사람.
* **일반 사용자:** 북마크한 웹사이트나 가지고 있는 문서의 내용을 쉽게 확인하고 싶은 모든 사람.

### 2. 사용자 스토리 및 핵심 기능 (User Stories & Features)

사용자가 우리 서비스를 통해 얻고자 하는 가치를 중심으로 기능을 정의합니다.

#### 2.1. Epic: URL을 이미지로 변환하기
* **As a** 블로거, **I want to** 내가 공유할 기사의 URL을 입력해서 웹사이트의 모습을 그대로 담은 이미지를 만들고 싶다. **So that** 내 블로그 포스팅이 더 풍부해 보이고 클릭률을 높일 수 있다.

| **기능 ID** | **기능명** | **상세 설명** | **우선순위** | **비고** |
| :--- | :--- | :--- | :--- | :--- |
| F-001 | **URL 입력 필드** | 사용자가 이미지를 생성할 웹사이트의 전체 URL을 입력할 수 있는 텍스트 필드를 제공한다. (예: [https://www.google.com](https://www.google.com)) | **Must-have** | |
| F-002 | **이미지 생성 버튼** | 입력된 URL의 웹페이지를 캡처하여 이미지 생성을 시작하는 버튼이다. | **Must-have** | |
| F-003 | **생성된 이미지 미리보기** | 변환이 완료된 이미지를 웹페이지에서 바로 보여준다. | **Must-have** | |
| F-004 | **이미지 다운로드** | 생성된 이미지를 사용자의 컴퓨터에 저장할 수 있는 다운로드 버튼을 제공한다. (PNG, JPG 형식 지원) | **Must-have** | |
| F-005 | **캡처 디바이스 선택** | 데스크톱, 태블릿, 모바일 등 다양한 화면 크기로 캡처할 수 있는 옵션을 제공한다. | **Should-have** | MVP 이후 고려 |
| F-006 | **전체 페이지 캡처 옵션** | 현재 보이는 화면뿐만 아니라, 스크롤을 내려야 보이는 전체 페이지를 캡처하는 옵션을 제공한다. | **Could-have** | |

#### 2.2. Epic: 파일을 이미지로 변환하기
* **As a** 대학생, **I want to** 내 리포트(PDF, DOC) 파일을 업로드해서 첫 페이지를 이미지로 만들고 싶다. **So that** 여러 버전의 리포트 파일을 시각적으로 쉽게 구분할 수 있다.

| **기능 ID** | **기능명** | **상세 설명** | **우선순위** | **비고** |
| :--- | :--- | :--- | :--- | :--- |
| F-007 | **파일 업로드 기능** | 사용자가 자신의 컴퓨터에서 HTML, PDF, DOC, MD 파일을 선택하여 업로드할 수 있는 인터페이스를 제공한다. (드래그 앤 드롭 지원) | **Must-have** | |
| F-008 | **파일 형식 자동 감지** | 업로드된 파일의 확장자를 통해 형식을 자동으로 인식하고, 그에 맞는 변환 로직을 실행한다. | **Must-have** | |
| F-009 | **파일 변환 및 미리보기** | 업로드된 파일의 첫 페이지를 이미지로 변환하여 화면에 보여준다. | **Must-have** | |
| F-010 | **지원 파일 형식 확장** | 향후 PPT, HWP 등 더 다양한 문서 파일 형식을 지원할 수 있도록 구조를 설계한다. | **Could-have** | |

### 3. 기술 스택 및 아키텍처 제안 (Tech Stack & Architecture)

주니어 개발자가 비교적 쉽게 접근하고 빠르게 프로토타입을 만들 수 있는 기술 스택을 제안합니다.

#### 3.1. 추천 기술 스택
* **프론트엔드 (Frontend):**
    * **HTML, CSS, JavaScript (Vanilla JS):** 별도의 프레임워크 없이 기본 기술만으로도 충분히 구현 가능하며, 학습 곡선이 낮습니다.
    * **(대안) React, Vue.js:** 컴포넌트 기반 개발에 익숙하다면 생산성을 높일 수 있습니다.
* **백엔드 (Backend):**
    * **Node.js & Express:** JavaScript에 익숙한 주니어 개발자가 프론트엔드와 동일한 언어로 개발할 수 있어 진입 장벽이 낮습니다. 비동기 처리에 강점이 있어 이미지 변환과 같은 작업에 유리합니다.
* **핵심 라이브러리/도구:**
    * **웹사이트/HTML 캡처:** **Puppeteer** 또는 **Playwright**. 이들은 헤드리스(Headless) 브라우저를 제어하는 라이브러리로, 실제 브라우저가 렌더링하는 것과 동일한 결과물의 스크린샷을 찍을 수 있어 가장 확실한 방법입니다. 
    * **PDF 변환:** **pdf-lib**, **PDF.js**. PDF 파일의 특정 페이지를 렌더링하여 이미지로 변환하는 데 사용됩니다.
    * **DOC(Word) 변환:** **LibreOffice (Headless mode)** 또는 **Pandoc**. 서버에 LibreOffice를 설치하고 커맨드 라인 명령어를 통해 DOC 파일을 PDF로 먼저 변환한 후, PDF를 다시 이미지로 변환하는 방식이 일반적입니다. (구현 난이도가 다소 높음)
    * **MD(Markdown) 변환:** **marked**, **showdown**. Markdown 텍스트를 HTML로 변환한 후, Puppeteer를 이용해 해당 HTML을 이미지로 캡처합니다.

#### 3.2. 고수준 아키텍처 (High-level Architecture)


1.  **사용자 (Client):** 웹 브라우저를 통해 서비스에 접속하고 URL을 입력하거나 파일을 업로드합니다.
2.  **웹 서버 (Web Server - Node.js/Express):** 사용자의 요청(Request)을 받습니다.
    * URL 입력 요청: URL을 받아 Puppeteer 모듈에 전달합니다.
    * 파일 업로드 요청: 업로드된 파일을 임시 저장하고, 파일 종류에 따라 적절한 변환 모듈(PDF, DOC, MD 처리기)에 전달합니다.
3.  **이미지 변환 엔진 (Conversion Engine):**
    * **Puppeteer:** 전달받은 URL에 접속하여 페이지 렌더링이 끝나면 스크린샷을 찍어 이미지 파일로 생성합니다.
    * **파일 처리기:** 각 파일 형식에 맞는 라이브러리를 사용하여 파일을 이미지로 변환합니다.
4.  **응답 (Response):** 웹 서버는 생성된 이미지 파일을 사용자에게 전송하여 화면에 보여주고, 다운로드할 수 있도록 합니다.

### 4. 개발 로드맵 (Roadmap)

#### Phase 1: MVP (최소 기능 제품) 개발 (1~2개월)
가장 핵심적인 기능에 집중하여 빠르게 시장에 선보이는 것을 목표로 합니다.

* **목표:** 사용자가 URL을 입력하고, 웹사이트의 첫 화면을 이미지로 받아 다운로드할 수 있다.
* **주요 기능 구현:**
    * `F-001`: URL 입력 필드
    * `F-002`: 이미지 생성 버튼
    * `F-003`: 생성된 이미지 미리보기
    * `F-004`: 이미지 다운로드 (PNG 형식 우선 지원)
    * 간단하고 직관적인 UI/UX 디자인

#### Phase 2: 파일 지원 기능 추가 (MVP 이후 1개월)
* **목표:** 사용자가 PDF와 Markdown 파일을 업로드하여 이미지로 변환할 수 있다.
* **주요 기능 구현:**
    * `F-007`: 파일 업로드 기능 (PDF, MD 우선 지원)
    * `F-008`: 파일 형식 자동 감지
    * `F-009`: 파일 변환 및 미리보기

#### Phase 3: 기능 고도화 및 안정화 (Phase 2 이후 2개월)
* **목표:** 사용자 편의성을 높이고 더 많은 파일 형식을 지원한다.
* **주요 기능 구현:**
    * `F-005`: 캡처 디바이스 선택 기능 추가
    * DOC 파일 변환 기능 추가 (서버 환경 설정 필요)
    * 변환 속도 최적화 및 오류 처리 강화
    * 사용자 피드백을 반영한 UI 개선

### 5. 비기능적 요구사항 (Non-functional Requirements)

* **성능 (Performance):** 이미지 변환 요청 후 10초 이내에 결과를 보여주는 것을 목표로 한다. (네트워크 상태 및 대상 웹페이지 복잡도에 따라 달라질 수 있음)
* **보안 (Security):** 사용자가 입력한 URL이나 업로드한 파일은 이미지 변환 목적 외에는 사용되거나 저장되어서는 안 된다. 변환이 완료되면 즉시 서버에서 삭제되어야 한다.
* **사용성 (Usability):** IT 지식이 없는 사용자도 설명 없이 바로 사용할 수 있도록 인터페이스가 직관적이고 쉬워야 한다.
* **확장성 (Scalability):** 향후 사용자가 증가하거나 지원하는 파일 형식이 늘어날 경우, 서버 구조를 유연하게 확장할 수 있도록 설계한다. (예: 이미지 변환 작업을 별도의 워커(Worker) 프로세스로 분리)

이 PRD 문서를 바탕으로 프로젝트의 큰 그림을 이해하고, 세부적인 기능 개발을 순차적으로 진행해 나가시길 바랍니다. 개발 과정에서 궁금한 점이나 어려운 부분이 있다면 언제든지 다시 질문해주세요! 성공적인 프로젝트 개발을 응원합니다. 💪
